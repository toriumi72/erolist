# docs/ README

このディレクトリは **サービスの要件・仕様・設計ドキュメント**を保存する場所です。
実装に入る前に「何を作るか」をここに残し、後から見返せる状態にします。

## ここに置くもの

- **要件定義 / 仕様書**（機能追加のたびに 1 つ作る）
- 画面仕様（ワイヤー、画面遷移、文言、バリデーション）
- データ設計（DB スキーマ、ER 図、インデックス方針、RLS 方針）
- ADR（Architecture Decision Record: 重要な技術判断の理由）
- 運用ルール（命名規則、ランキング計算の定義、スコアリング基準など）

## ここに置かないもの

- 実装コード（→ `app/`, `components/`, `services/` など）
- 生成物・バイナリ（巨大ファイルは避ける）
- 秘密情報（API キー等）

## 推奨する運用（最小）

- 機能を作る前に `docs/specs/` に 1 ファイル追加する
- 実装が進んで仕様が変わったら、その差分もドキュメントに反映する

## 推奨ディレクトリ構成（必要になったら作る）

- `docs/specs/`：機能要件・仕様（例: `site-list.md`, `ranking.md`）
- `docs/adr/`：技術判断ログ（例: `0001-use-server-actions.md`）
- `docs/db/`：DB 設計（例: `schema.md`, `rls-policies.md`）
- `docs/ui/`：画面仕様（例: `screens.md`）

## 仕様書テンプレ（コピペ用）

以下の章立てで書くとブレにくいです。

- 目的 / 背景
- 期待するユーザー体験（どう便利になるか）
- 仕様（画面・操作・表示）
- データ（入力/出力、保存するもの、制約）
- ランキング等の定義（計算式・優先順位）
- エッジケース / エラー時の挙動
- 非機能（パフォーマンス、セキュリティ、運用）
- 受け入れ条件（これが満たせたら完了）
